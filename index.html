제안 포인트

1. 앱 구조 — 두 가지 옵션 제안

본 프로젝트는 B2B 디자이너 앱과 B2C 고객 앱의 사용자 흐름·기능·노출 메뉴가 명확히 다르므로, 두 가지 앱 구조 옵션을 제시합니다. 미팅 시 운영 정책·스토어 전략(검색 노출, 사용자 혼선 등)을 고려하여 선택하실 수 있도록 양쪽 모두 구현 경험을 확보하고 있습니다.

  • [옵션 A — 기본 제안] 별도 앱 2종 — 디자이너 앱(B2B)과 고객 앱(B2C)을 별도 스토어 등록 분리. 각 앱이 본인 역할에만 최적화되어 검색 노출이 명확하고 디자이너 전용 기능(예약가능시간 설정·정산·쿠폰 발급 등)을 모바일 첫 화면부터 노출 가능. 본 견적서·일정표는 이 옵션 기준.
  • [옵션 B — 협의 가능] 공용 단일 앱 + 권한 분리 UI — 단일 Flutter 앱에서 로그인 시 역할(디자이너/고객) 판정 후 홈 화면·하단 탭바·메뉴 구성을 동적으로 분기. 디자이너 전용 메뉴(스케줄·정산 등)는 디자이너 계정에만 노출, 고객 전용(예약·웨이팅 등)은 고객 계정에만 노출. 장점: 빌드·심사·유지보수 1회로 통합, 코드 재활용 극대화. 단점: 첫 화면 분기 로직 복잡, 스토어 카테고리 단일화로 검색 최적화 제약. 옵션 B 선택 시 약 5~7 M/D, 200~300만원 절감 가능.

2. 4대 핵심 비즈니스 로직 — 검증된 패턴으로 정확하게 구현

  • 양방향 웨이팅: PostgreSQL SELECT FOR UPDATE SKIP LOCKED + Redis 분산락(Redlock) 조합으로 다중 동시 수락 상황에서도 단일 결제만 통과시키는 동시성 제어. 서버 시간 기준 15분 타이머는 백그라운드 워커(BullMQ)가 만료 시점에 자동으로 다음 큐 1순위에게 이전 처리.
  • 환불·정산 회계: 모든 거래를 복식부기 방식의 ledger 테이블에 기록(차변/대변), 이벤트 소싱으로 모든 변경을 Audit Log로 보존. 24h/12h/6h 차등 환불은 정책 테이블에서 동적으로 조회하여 정책 변경 시 코드 배포 없이 운영 가능.
  • QR 디퍼드 딥링크 — 자체 구현: 외부 솔루션(Branch.io/AppsFlyer) 없이 자체 QR + 파라미터 URL 방식으로 구현. https://mydesigner.com/d/abc123 URL을 QR로 만들어, 앱 설치 시 Android Install Referrer API(공식·무료·정확도 99%+)로 designer_id 자동 매칭, iOS는 Universal Link + 첫 실행 환영 화면 폴백. 솔루션 비용·종속성·사망 리스크(Firebase Dynamic Links 종료 사례) 회피. MAU 100K 돌파 시 솔루션 도입 재검토.
  • 커머스 정산: 결제 완료 시 디자이너/플랫폼 ledger에 50:50 분개를 동시 기록, 월 단위로 디자이너에게 송금 요청 가능 잔액으로 집계.

3. 인스타그램형 단골 피드 메인 — 유사 매칭 플랫폼 화면 노하우 직접 활용

대면 미팅에서 확인된 메인 화면(상단 단골 디자이너 아바타 카드 + 하단 디자이너 작업 사진 쓰레드 피드) 구조는 디지털 명함 매칭 플랫폼에서 구현한 "근거리 추천 사용자 + 비즈니스 소셜 피드" 화면 구조와 동일 패턴입니다. 가로 스크롤 아바타 리스트, 무한 스크롤 피드 페이지네이션, 이미지 캐싱 및 프리페치(Flutter cached_network_image + flutter_bloc 스트림 처리), 좋아요·예약 즉시 전환 CTA 등 핵심 구현 패턴을 그대로 재사용하여 메인 화면을 안정적으로 빠르게 완성합니다.

4. 디자이너 간 데이터 격리 — RLS + 멀티테넌트 모델

모든 고객 데이터(예약, 처방전, 쿠폰, 포인트)는 designer_id 외래키를 가지며, PostgreSQL Row-Level Security(RLS) 정책으로 디자이너 본인 데이터만 조회·수정 가능하도록 DB 레벨에서 강제합니다. 애플리케이션 코드 버그가 있어도 다른 디자이너의 데이터 노출이 원천 차단됩니다. 권한은 CASL 기반 RBAC로 디자이너/관리자/슈퍼관리자 역할 분리.

5. 관리자 페이지 — B2B2C SaaS 운영 콘솔 패턴 재활용

본 프로젝트와 동일한 B2B2C 구조의 헬스케어 SaaS(시니어 주간보호센터 통합 관리 플랫폼)에서 구축한 운영자 콘솔 구조를 재활용합니다. 사이드바 네비게이션 + 통계 카드 + 가입 승인 워크플로우 + 정산 잔액 대시보드 + 감사 로그 패턴이 본 프로젝트의 디자이너 가입 승인·정산·포인트 부채 관리 요구사항과 1:1 매칭됩니다.

6. Flutter 시연 코드 활용 — 폐기 없이 점진 고도화

제공된 단일 main.dart 시연 코드를 분석하여 (1) 화면별로 feature 디렉토리 분리, (2) Riverpod Provider/AsyncNotifier 패턴으로 상태관리 도입, (3) Repository 레이어 추가로 API 연동 준비, (4) 누락된 화면 30%를 디자인 가이드 기준으로 보완. 기존 UI는 최대한 보존하여 기획자가 검토한 화면 흐름이 그대로 유지됩니다.

7. 미확정 정책 사전 협의 — 1주차 정책 결정 워크숍

15분 웨이팅 알림 자동·수동 발송 여부, 환불 정책 세부 시점, 위약금 산정 기준, 앱 구조(옵션 A/B) 선택 등 미확정 항목을 1주차 정책 결정 워크숍에서 일괄 협의·확정합니다. 정책 결정 사항은 정책 문서로 정리되어 개발 명세서 v3.1과 함께 단일 진실 공급원(Single Source of Truth)으로 관리됩니다.

Phase 1 MVP 범위의 기능 요구사항을 정의합니다. 상세 명세서 v3.1은 미팅 시 제공받은 후 본 PRD를 기준으로 정합화합니다.

1. 프로젝트 배경 및 목적

미용 디자이너의 단골 관리, 자생력 확보, 지속 가능한 창업 인프라를 통합 제공하는 SaaS 플랫폼. 단순한 예약 마켓플레이스가 아닌 디자이너의 단골 고객 자산화 + 시술 후 커머스 결합형 비즈니스 지원 플랫폼을 목표로 하며, 디자이너가 본인의 단골을 직접 데려와 운영하는 B2B2C 모델입니다. Phase 1 MVP는 4대 핵심 차별화 기능(양방향 웨이팅, 홈케어 커머스, 디자이너 전용 쿠폰, 단골 자동 매칭)을 시장에 빠르게 출시하는 것이 목표입니다.

2.1 양방향 웨이팅 시스템 (명세서 5.1, 11.3)

작동 흐름 (사용자 관점)

  1. 고객이 마감 시간대 슬롯에 웨이팅 신청 → 대기열에 등록 (FIFO 순서)
  2. 예약자가 취소 발생 → 대기열 1순위 고객에게 15분 타임어택 푸시 전송
  3. 1순위 고객이 15분 내 결제 완료 → 슬롯 확정, 대기열에서 제거
  4. 1순위 고객이 15분 내 결제하지 않음 → 자동으로 2순위로 이전, 15분 타이머 다시 시작
  5. 대기열이 소진되거나 누군가 결제 완료할 때까지 반복

기술 구현 방식

  • 대기열 저장: Redis FIFO 큐(LIST 자료구조)에 사용자 ID를 순서대로 저장. 푸시·자동 이전 시 1순위를 원자적으로 추출(LPOP). Redis 특성상 큐 read/write 속도가 빠르고 순서 보장이 명확
  • 결제 확정 시 DB 기록: 사용자가 결제 완료하면 PostgreSQL reservations 테이블에 예약 INSERT + 대기열에서 해당 사용자 제거를 단일 트랜잭션으로 처리. 슬롯 점유는 결제 확정 시점에 DB에 영구 기록
  • 15분 타이머: Redis TTL 키 활용. 1순위에게 푸시 발송 시 SET active_waiting:slot_id user_id EX 900 (900초). TTL 만료 이벤트 감지(Redis Keyspace Notifications) 또는 1분 주기 폴링으로 만료된 슬롯을 자동으로 다음 사용자에게 이전
  • 다중 사용자 동시 결제 시도 동시성 제어: 결제 확정 트랜잭션에서 PostgreSQL SELECT FOR UPDATE로 슬롯 행 잠금. 동시에 결제 버튼을 눌러도 트랜잭션이 순차 처리되어 단일 고객만 통과
  • 중복 요청 방어: 결제·예약 수락 API에 멱등성 키(Idempotency-Key) 헤더 적용. 네트워크 재시도·중복 클릭에도 단일 처리 보장
  • 상태 추적: 큐 등록 → 푸시 발송 → 수락/거절/만료 → 결제 완료 전 과정을 audit_logs에 기록 (분쟁 대응 및 정합성 검증용)
  • 알림: FCM 푸시 기본, 앱 비활성 사용자는 알림톡 fallback
  • 시간 기준: AWS/NCP 서버는 NTP가 자동 동기화되어 있어 서버 시각 기준으로 타이머 운영. 클라이언트 앱 타이머는 서버 응답의 절대 시각을 기준으로 동기화하여 클라이언트 시계 변조에 대비

자동/수동 발송 정책 제안

  • 주간(08:00~22:00): 자동 발송 — 취소 발생 즉시 1순위에게 푸시
  • 야간(22:00~08:00): 수동 발송 권장 — 사용자가 새벽에 푸시 받는 부담을 줄이기 위해, 디자이너가 다음 날 오전 확인 후 수동으로 발송 트리거
  • 위 기본값은 1주차 정책 워크숍에서 협의 후 확정

  • 환불 정책 테이블: 시점별 환불 비율을 DB 정책 테이블로 관리하여 정책 변경 시 코드 배포 없이 운영자 콘솔에서 수정 가능
  • 1시간 그레이스: 예약 시간 1시간 이내 1회에 한해 변경 가능 (변경 이력은 audit_logs 기록)
  • PortOne 환불 API + webhook: 환불 요청 시 PortOne 환불 API 호출, webhook 수신으로 환불 완료 확정 (중복 webhook 대비 멱등성 처리 적용)
  • 부분 환불: 현금 + 포인트 분할 환불을 단일 트랜잭션으로 처리하여 정합성 보장 (예: 70% 현금 + 30% 포인트)
  • 정산 정합성: 이미 디자이너에게 정산된 결제 환불 시 → 디자이너 잔액에서 차감 → 다음 정산 사이클 반영. 음수 잔액 발생 시 운영자 알림 + 다음 매출에서 자동 상쇄
  • 환불 실패 재시도: PG 오류·네트워크 오류 발생 시 재시도 큐로 등록(짧은 backoff 간격, 3회 시도). 최종 실패 시 운영자 콘솔 알림
  • 환불 영수증·세금계산서: PortOne이 환불 영수증 자동 발급, 세금계산서는 디자이너 사업자 정보 기반 자동 또는 수동 워크플로우
  • 분쟁 처리: 디자이너↔고객 분쟁 발생 시 운영자 콘솔에서 수동 환불 권한 + 분쟁 이력·증빙 기록 워크플로우 제공
  • audit_logs 보존: 1년 hot 보관 (운영 분석·분쟁 대응용). 결제 정보 자체는 PortOne이 PCI-DSS 인증으로 보관하여 전자상거래법 5년 의무를 충족

2.3 포인트 시스템 (명세서 5.13)

구현 방식

  • 포인트 거래 원장(ledger) 방식: 모든 적립·사용·만료·환불을 거래로 기록. 잔액은 ledger 합산으로 도출하여 캐싱(Redis)으로 조회 성능 확보. 회계·정산·분쟁 대응이 명확
  • 포인트 3종 분리: 활성 포인트 / 마케팅 포인트(가속·웰컴·이벤트) / 환불 포인트 — 출처·만료·세무 처리가 다르므로 분리 저장
  • FIFO 만료: 발급 순서대로 사용, 만료 30일 전·7일 전 2회 사전 알림(푸시 + 알림톡 fallback). 만료 처리는 매일 자정 cron job
  • 환전율: MVP는 1:1 통일 (다단계 환전 도입은 고도화 영역에서 협의)
  • 결제 시 사용 정책: 1결제당 최대 50% 포인트 사용, FIFO 차감 (정책 테이블 기반, 운영자 수정 가능)
  • 부정 사용 방지 (기본 사양):
    • 동일 디바이스 ID 다중 계정 가입 차단 (기본 디바이스 핑거프린팅)
    • 일 20회 사용 한도 (임계값 카운터)
    • 신고 시 자동 차단 + 운영자 검토 큐
    • 적발 시 활성 포인트 차감 + 계정 정지 + audit_logs 기록
    • (고도화에서 룰 엔진 추가·이상 패턴 자동 탐지 등 단계적 강화 가능)
  • 포인트 부채 관리: 발급된 미사용 포인트 = 회사 부채로 인식. 관리자 콘솔에 포인트 부채 잔액 대시보드 제공 (명세서 11.6 반영)

2.4 실시간 동기화 (명세서 5.11)

알림 처리

  • 웨이팅 1순위 알림, 예약 확정·취소 알림 등은 FCM 푸시로 즉시 도달 (앱 백그라운드 포함)
  • 푸시 전송은 비동기 큐 기반으로 처리하여 트래픽 폭증 시에도 안정적 운영
  • 채널 fallback: FCM 실패 시 알림톡 → SMS(결제·긴급 한정) → 이메일 순으로 진행
  • 푸시 토큰 만료 자동 감지·갱신, 만료 토큰 자동 제거
  • 푸시 발송 결과는 audit_logs 기록 (도달률 모니터링 + 클레임 대응)
  • 푸시 클릭 시 앱 내 특정 화면으로 직접 이동(딥링크 처리)

관리자 대시보드

  • 30~60초 폴링으로 통계 갱신 (별도 WebSocket 인프라 없이 충분히 운영 가능)

결제 webhook 처리

  • PortOne webhook 수신 시 멱등성 키 + DB unique constraint로 중복 호출 방어 (PG가 webhook 재전송하는 경우 대응)
  • 주문 상태 업데이트 → 사용자·디자이너 알림 발송
  • 백엔드에서 PortOne API로 결제 결과를 한 번 더 검증하여 webhook 신뢰성 보강 (위변조 방어)
  • Webhook 처리 실패 시 PortOne의 자동 재시도(1분 → 5분 → 15분 → 1시간 backoff)와 연계

결제 정합성 검증

  • 매일 03:00 야간 배치로 PortOne 정산 데이터와 자체 DB 정산 ledger의 일치 여부 검증
  • 불일치 발견 시 Slack 알림 + 운영자 콘솔 표시. 운영자가 사유 기록 후 수동 조정(audit_logs)

디자이너 데이터 격리

  • 모든 디자이너 종속 테이블에 designer_id 외래키 + 인덱스 적용
  • 애플리케이션 레이어에서 ORM 인터셉터로 designer_id 강제 (서비스 레이어 일관성 확보)
  • 데이터 격리 강화가 필요한 시점에 PostgreSQL Row-Level Security 정책을 추가 적용하는 방향으로 점진 강화 가능. 처음부터 RLS 적용도 협의 가능
  • Supabase 미사용, 자체 PostgreSQL 운영으로 인프라 종속성 회피

2.5 결제·정산 핵심 (명세서 5.12)

결제 흐름

  • 클라이언트 → PortOne SDK → PG 결제 페이지 → 결제 완료
  • 백엔드는 PortOne webhook을 수신하면 멱등성 검증 후 주문 상태 업데이트
  • 추가로 백엔드에서 PortOne API로 결제 결과를 재검증하여 위변조 방어
  • 결제 정보는 PortOne에 PCI-DSS 인증으로 보관되어 자체 DB에는 imp_uid(결제 식별 토큰)만 저장

결제 우선순위 및 정책 스냅샷

  • 결제 시 차감 순서: 포인트 → 쿠폰 → 카드(잔액). 정책 테이블 기반으로 운영자 수정 가능
  • 결제 시점 정책 스냅샷(policy_snapshot_at_order) 컬럼 저장 — 정책 변경 시 소급 적용 방지

마진 비율

  • MVP 통일 5:5
  • (고도화에서 디자이너 등급별 4:6 / 5.5:4.5 / 6:4 분기 도입 가능)
  • 결제 시점 등급 스냅샷(designer_tier_at_order) 컬럼 저장

알림 채널 정책

  • 단계적 fallback 방식 (동시 발송 X): 푸시(앱 활성) → 알림톡(앱 비활성) → SMS(결제·긴급 한정) → 이메일(영수증·공지)
  • 각 채널 도달 실패 시 다음 채널로 자동 진행. 채널별 비용 절감 + 도달률 균형 확보

정산 처리

  • 정산 거래 원장(ledger) 방식 — 매출 / 플랫폼 수수료 / 디자이너 수익 / 환불 / 위약금 / 송금을 거래로 기록하여 회계 정합성과 세무 대응 확보
  • 매일 03:00 cron job으로 전일 정산 데이터 집계 + 디자이너별 잔액 업데이트
  • 디자이너 송금 요청 → 운영자 콘솔에서 검토 → 송금 처리 (수동 승인 후 자동 이체 또는 수동 이체)
  • 월간 정산 보고서 자동 생성 (PDF + CSV, 디자이너 다운로드)

세금계산서·현금영수증

  • 디자이너 사업자 정보 기반 세금계산서 자동 발행 워크플로우 (PortOne 부가 서비스 또는 빌터 연동)
  • 고객 현금영수증 자동 발행 (PortOne 부가 서비스)
  • 분기·연 단위 세무 신고 자료 CSV 자동 출력

매출 정합성 검증

  • 매일 03:00 야간 배치로 PortOne 정산 데이터와 자체 정산 ledger의 일치 여부 검증
  • 불일치 발견 시 Slack 알림 + 운영자 콘솔에 표시. 운영자가 사유 기록 후 수동 조정

보안

  • 카드 정보: PortOne(PCI-DSS) 보관, 자체 DB는 결제 토큰만 저장
  • 개인정보(이름·휴대폰·생년월일·계좌번호·사업자번호): AES-256 컬럼 암호화 적용
  • 환경 변수·API 키: AWS Secrets Manager로 관리
  • (개인정보 컬럼 암호화 키 관리에 AWS KMS 도입 가능 — 옵션)

2.6 QR 디퍼드 딥링크 (명세서 5.6)

솔루션 선택 — 자체 구현 제안

외부 솔루션(Branch.io 등) 도입 없이 자체 구현으로 진행 가능하다고 판단됩니다. 자체 QR + Universal Link/App Link + Android Install Referrer + 휴대폰 번호 매칭 조합으로 본 서비스가 필요로 하는 매칭 정확도를 확보할 수 있고, 외부 솔루션 종속성·운영 비용·서비스 종료 리스크(Firebase Dynamic Links 2025.08 종료 사례)도 회피할 수 있습니다. 향후 MAU·매칭 손실률 추이를 보고 외부 솔루션 도입을 재검토하는 방향이 합리적입니다.

구현 아키텍처

  1. QR 생성
    • 디자이너 가입 시 고유 식별자(slug) 발급
    • QR URL 형식: https://mydesigner.com/d/{slug}?utm_source=qr...
    • UTM 파라미터로 유입 분석 지원
    • QR 만료 시간(예: 90일) + 인증된 디자이너만 생성 + 사용 추적 audit_logs
  2. 딥링크 도메인 검증 파일 호스팅 (필수)
    • iOS: apple-app-site-association 호스팅 (Universal Link 작동 조건)
    • Android: assetlinks.json 호스팅 (App Link 작동 조건)
  3. URL 라우팅
    • 앱 설치 + 일반 브라우저: Universal Link/App Link 자동 작동 → 앱이 designer_id 받아 매칭
    • 앱 미설치: Play Store / App Store 다운로드 페이지 리디렉트
    • 카카오톡 인앱 브라우저: User-Agent 감지 → 폴백 페이지에서 "Chrome/Safari로 열기" 안내. Android는 intent:// 스킴으로 외부 브라우저 직접 호출
  4. 디퍼드 매칭 (앱 미설치 → 설치 후 매칭)
    • Android: Google Play Install Referrer API (공식·무료, 정확도 99%+). Play Store URL에 designer_id가 referrer로 자동 첨부되며, 앱 첫 실행 시 자동 수신하여 매칭
    • iOS (다중 폴백 전략)
      • (a) 휴대폰 번호 매칭: 디자이너가 매장에서 단골 고객의 휴대폰 번호를 사전 등록 → 사용자가 가입 시 휴대폰이 일치하면 자동 단골 매칭. 비즈니스 흐름에 자연스럽게 통합되며 정확도가 높음
      • (b) 환영 화면 추천 코드 입력 — 앱 첫 실행 시 "어떤 디자이너의 추천으로 오셨나요?" 화면 + 디자이너 검색
      • (c) Apple Search Ads API (광고 캠페인 한정 사용)
  5. 매칭 정확도 모니터링
    • 매칭 성공/실패 로그를 운영자 콘솔 대시보드에 표시
    • 매칭 손실률이 일정 임계값을 초과하면 외부 솔루션 도입을 재검토
    • 디자이너별 QR 유입 통계 제공
  6. 예상 매칭 정확도: Android 99%+ / iOS 95% 내외 (휴대폰 매칭 + 환영 화면 폴백 포함)
  7. 향후 솔루션 도입 검토 시점: MAU 증가에 따른 iOS 매칭 손실률이 비즈니스 손해로 누적되는 시점에 Branch.io 또는 Adjust 도입 재검토 (자체 구현 폴백은 유지)

2.7 인프라·보안·데이터 보존 (명세서 5.6.4, 6.3, 9)

DB 인프라

  • RDS PostgreSQL + 자동 백업 일 1회 + 7일 보관 (기본 설정)
  • 가용성 강화 옵션으로 Multi-AZ 적용 가능 (단일 인스턴스 대비 비용 약 1.5~2배 증가). MVP 출시 단계에서는 단일 인스턴스 + 자동 백업으로 시작 후 트래픽 안정화 시점에 Multi-AZ 전환을 권장 드리며, 처음부터 Multi-AZ 적용도 협의 가능합니다
  • 분기별 수동 스냅샷 + S3 장기 보관
  • 분기 1회 백업 복구 절차 검증 (운영 매뉴얼에 절차 명시)

디자이너 데이터 격리

  • 모든 디자이너 종속 테이블에 designer_id 외래키 + 인덱스 적용
  • 애플리케이션 레이어(ORM 인터셉터)에서 designer_id 강제 — 서비스 코드 일관성·디버깅 편의성 확보
  • 데이터 격리 강화가 필요한 시점에 PostgreSQL Row-Level Security 정책을 단계적으로 추가 적용. 처음부터 RLS 적용을 원하시면 협의 가능합니다
  • Supabase 미사용, 자체 PostgreSQL 운영으로 인프라 종속성 회피

개인정보 보호

  • AES-256 컬럼 암호화: 이름·휴대폰·생년월일·계좌번호·사업자번호
  • 암호화 키는 환경 변수 또는 AWS Secrets Manager로 관리 (강화 필요 시 AWS KMS 도입)
  • 로그 자동 마스킹 (예: 휴대폰 010-****-1234, 이메일 a***@b.com)
  • 운영자 콘솔에서도 권한 범위 외 개인정보는 마스킹 표시
  • 회원 탈퇴 후 보존: 결제 관련 5년 보관(전자상거래법), 그 외 즉시 파기 또는 익명화 (PIPA 준수)

데이터 보존

  • audit_logs: 1년 hot 보관 (운영 분석·분쟁 대응용)
  • 결제 정보: PortOne에 PCI-DSS 인증으로 보관 → 전자상거래법 5년 의무 충족
  • 회원 정보: 전자상거래법 기준 결제 관련 5년 보관, 그 외 익명화 또는 파기
