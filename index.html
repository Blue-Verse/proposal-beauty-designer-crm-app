핵심 요구사항 분석

  • B2B 디자이너 앱 — 본인인증, 사업자/계좌 등록, 스케줄·예약가능시간 관리, 골든타임 리커버리, 스텔스 차단, 처방전·홈케어 가이드 발급, 디자이너 전용 쿠폰, 단골 팬덤, 정산금 조회
  • B2C 고객 앱 — 디자이너 탐색·예약, 쿠폰·포인트 복합 결제, 마감 시간대 웨이팅 등록, 마이룸(포인트/쿠폰/홈케어가이드 보관함)
  • 양방향 웨이팅 동시성 제어 — 취소 발생 시 큐 1순위에게 15분 타임어택 푸시, 다중 수락 시 DB Lock + Redis 분산락으로 단일 결제 보장
  • 환불·위약금 회계 — 24h/12h/6h 차등 환불 정책(현금/포인트), 디자이너 위약금 정산, Audit Log 기반 트랜잭션 무결성
  • QR 디퍼드 딥링크 — 디자이너 고유 QR 스캔 → 앱 설치 유도 → 첫 실행 시 단골 자동 매칭, 카카오톡 인앱 브라우저 우회 필수
  • 커머스 정산 — 홈케어 제품 판매 시 디자이너/플랫폼 5:5 마진 분배 자동화
  • 관리자 웹 — 디자이너 가입 승인, 고객 차단·신고 처리, 카테고리 관리, 정산 잔액 조회, 포인트 부채 대시보드
  • 외부 연동 — 포트원(결제), FCM(푸시), 알림톡, Branch.io(딥링크), 본인인증 PG
  • 보안 — JWT 인증, RBAC + Row-Level Security, 개인정보(이름·휴대폰) 암호화 저장
  • 제공 자산 활용 — Flutter 시연 코드(UI 70%) 분석 후 Riverpod 기반 상태관리 고도화 및 누락 화면 보완

제안 포인트

1. 4대 핵심 비즈니스 로직 — 검증된 패턴으로 정확하게 구현

  • 양방향 웨이팅: PostgreSQL SELECT FOR UPDATE SKIP LOCKED + Redis 분산락(Redlock) 조합으로 다중 동시 수락 상황에서도 단일 결제만 통과시키는 동시성 제어. 서버 시간 기준 15분 타이머는 백그라운드 워커(BullMQ)가 만료 시점에 자동으로 다음 큐 1순위에게 이전 처리.
  • 환불·정산 회계: 모든 거래를 복식부기 방식의 ledger 테이블에 기록(차변/대변), 이벤트 소싱으로 모든 변경을 Audit Log로 보존. 24h/12h/6h 차등 환불은 정책 테이블에서 동적으로 조회하여 정책 변경 시 코드 배포 없이 운영 가능.
  • QR 디퍼드 딥링크: Branch.io 추천 — Universal Link/App Link + 자체 호스팅 폴백 페이지로 카카오톡 인앱 브라우저 우회. 디자이너 ID는 QR URL의 unique slug + Branch deep_link_data에 이중 인코딩하여 어떤 환경에서도 매칭 정보 유실 없음.
  • 커머스 정산: 결제 완료 시 디자이너/플랫폼 ledger에 50:50 분개를 동시 기록, 월 단위로 디자이너에게 송금 요청 가능 잔액으로 집계.

2. 디자이너 간 데이터 격리 — RLS + 멀티테넌트 모델

모든 고객 데이터(예약, 처방전, 쿠폰, 포인트)는 designer_id 외래키를 가지며, PostgreSQL Row-Level Security(RLS) 정책으로 디자이너 본인 데이터만 조회·수정 가능하도록 DB 레벨에서 강제합니다. 애플리케이션 코드 버그가 있어도 다른 디자이너의 데이터 노출이 원천 차단됩니다. 권한은 CASL 기반 RBAC로 디자이너/관리자/슈퍼관리자 역할 분리.

3. Flutter 시연 코드 활용 — 폐기 없이 점진 고도화

제공된 단일 main.dart 시연 코드를 분석하여 (1) 화면별로 feature 디렉토리 분리, (2) Riverpod Provider/AsyncNotifier 패턴으로 상태관리 도입, (3) Repository 레이어 추가로 API 연동 준비, (4) 누락된 화면 30%를 디자인 가이드 기준으로 보완. 기존 UI는 최대한 보존하여 기획자가 검토한 화면 흐름이 그대로 유지됩니다.

4. 미확정 정책 사전 협의 — 1주차 정책 결정 워크숍

15분 웨이팅 알림 자동·수동 발송 여부, 환불 정책 세부 시점, 위약금 산정 기준 등 미확정 항목을 1주차 정책 결정 워크숍에서 일괄 협의·확정합니다. 정책 결정 사항은 정책 문서로 정리되어 개발 명세서 v3.1과 함께 단일 진실 공급원(Single Source of Truth)으로 관리됩니다.

5. 유사 프로젝트 수행 경험으로 효율적 진행

예약·결제 시스템 + 프랜차이즈 멀티테넌트 + 정산 회계 + Flutter 멀티플랫폼 앱 도메인을 모두 커버하는 유사 프로젝트 수행 경험을 보유하여, 본 MVP의 4대 핵심 차별화 기능 구현에 필요한 기술적 의사결정을 빠르게 내릴 수 있습니다. (상세는 "유사 프로젝트 경험" 페이지 참조)

PRD (Product Requirements Document)

Phase 1 MVP 범위의 기능 요구사항을 정의합니다. 상세 명세서 v3.1은 미팅 시 제공받은 후 본 PRD를 기준으로 정합화합니다.

1. 프로젝트 배경 및 목적

미용 디자이너의 단골 관리, 자생력 확보, 지속 가능한 창업 인프라를 통합 제공하는 SaaS 플랫폼. 단순한 예약 마켓플레이스가 아닌 디자이너의 단골 고객 자산화 + 시술 후 커머스 결합형 비즈니스 지원 플랫폼을 목표로 하며, 디자이너가 본인의 단골을 직접 데려와 운영하는 B2B2C 모델입니다. Phase 1 MVP는 4대 핵심 차별화 기능(양방향 웨이팅, 홈케어 커머스, 디자이너 전용 쿠폰, 단골 자동 매칭)을 시장에 빠르게 출시하는 것이 목표입니다.

3. 기능 요구사항

3.1 B2B 디자이너 앱

  • [필수] 소셜 로그인 (카카오/네이버/Apple/Google) + 본인인증 (PG 연동)
  • [필수] 사업자정보 및 정산 계좌 등록
  • [필수] 포트폴리오 등록 (사진 업로드, 시술 카테고리 태깅)
  • [필수] 스케줄 및 예약 가능 시간 설정 (요일별/날짜별)
  • [필수] 골든타임 리커버리 발송 (단골 자동 알림)
  • [필수] 스텔스 차단 (특정 고객 노출 차단)
  • [필수] 처방전 및 홈케어 가이드 발급
  • [필수] 디자이너 전용 쿠폰 발행소
  • [필수] 단골 팬덤 리스트 관리
  • [필수] 정산금 조회 및 수동 송금 신청

3.2 B2C 고객 앱

  • [필수] 소셜 로그인 (카카오/네이버/Apple/Google)
  • [필수] 디자이너 탐색 및 상세 프로필 조회
  • [필수] 시술/날짜/시간 선택 예약
  • [필수] 쿠폰 + 포인트 복합 적용 결제 (포트원)
  • [필수] 마감 시간대 웨이팅 등록
  • [필수] 15분 타임어택 결제 (큐 1순위 자동 푸시)
  • [필수] 예약 내역 조회/취소
  • [필수] 마이룸 (포인트/쿠폰/홈케어가이드 보관함)
  • [필수] QR 디퍼드 딥링크 첫 실행 단골 자동 매칭

3.3 핵심 비즈니스 로직

  • [필수] 양방향 웨이팅 — 취소 발생 시 큐 1순위에게 15분 타임어택 푸시(자동 발송 정책 1주차 확정), 미수락 시 자동 이전, 다중 수락 시 DB Lock + Redis 분산락으로 단일 결제 보장
  • [필수] 환불·위약금 회계 — 24h/12h/6h 차등 환불(현금/포인트), 디자이너 위약금 정산, 모든 변경은 Audit Log 기록
  • [필수] QR 디퍼드 딥링크 — Branch.io 기반 Universal Link/App Link, 카카오톡 인앱 브라우저 우회 폴백 페이지
  • [필수] 커머스 정산 — 홈케어 제품 판매 시 디자이너/플랫폼 5:5 마진 자동 분개

3.4 관리자 웹

  • [필수] 디자이너 가입 승인 워크플로우
  • [필수] 고객 차단/신고 처리
  • [필수] 시술 카테고리 관리 (CRUD)
  • [필수] 디자이너별 정산 잔액 조회
  • [필수] 포인트 부채 잔액 대시보드

5. 기술 제약사항

  • Flutter 시연 코드(단일 main.dart, UI 70%)를 기반으로 점진 고도화 — 폐기 후 재작성 금지
  • 카카오톡 인앱 브라우저 환경에서 QR 딥링크가 반드시 동작해야 함 (Universal Link 차단 회피 필수)
  • 웨이팅 시스템의 타이머는 클라이언트 시계가 아닌 서버 시계 기준
  • 본인인증은 KISA 인증 PG(NICE/KCB/Toss 본인인증) 사용
  • 결제 PG는 포트원(PortOne, 구 아임포트) 사용
  • 인프라는 AWS 또는 NCP — 미팅 시 클라이언트 선호 확인 후 결정
