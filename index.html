제안 포인트

1. 앱 구조 — 두 가지 옵션 제안

본 프로젝트는 B2B 디자이너 앱과 B2C 고객 앱의 사용자 흐름·기능·노출 메뉴가 명확히 다르므로, 두 가지 앱 구조 옵션을 제시합니다. 미팅 시 운영 정책·스토어 전략(검색 노출, 사용자 혼선 등)을 고려하여 선택하실 수 있도록 양쪽 모두 구현 경험을 확보하고 있습니다.

  • [옵션 A — 기본 제안] 별도 앱 2종 — 디자이너 앱(B2B)과 고객 앱(B2C)을 별도 스토어 등록 분리. 각 앱이 본인 역할에만 최적화되어 검색 노출이 명확하고 디자이너 전용 기능(예약가능시간 설정·정산·쿠폰 발급 등)을 모바일 첫 화면부터 노출 가능. 본 견적서·일정표는 이 옵션 기준.
  • [옵션 B — 협의 가능] 공용 단일 앱 + 권한 분리 UI — 단일 Flutter 앱에서 로그인 시 역할(디자이너/고객) 판정 후 홈 화면·하단 탭바·메뉴 구성을 동적으로 분기. 디자이너 전용 메뉴(스케줄·정산 등)는 디자이너 계정에만 노출, 고객 전용(예약·웨이팅 등)은 고객 계정에만 노출. 장점: 빌드·심사·유지보수 1회로 통합, 코드 재활용 극대화. 단점: 첫 화면 분기 로직 복잡, 스토어 카테고리 단일화로 검색 최적화 제약. 옵션 B 선택 시 약 5~7 M/D, 200~300만원 절감 가능.

2. 4대 핵심 비즈니스 로직 — 검증된 패턴으로 정확하게 구현

  • 양방향 웨이팅: 대기열은 Redis에 FIFO 순서로 관리하고, 결제 확정 시 DB에 영구 기록합니다. 여러 사용자가 동시에 결제를 시도해도 한 명만 통과되도록 잠금 처리하며, 15분 타이머가 만료되면 자동으로 다음 순위 사용자에게 이전됩니다.
  • 환불·정산 회계: 모든 거래(결제·환불·위약금·송금)를 거래 내역 테이블에 기록하여 추적·분쟁 대응에 활용합니다. 환불 정책(24h/12h/6h)은 운영자 콘솔에서 직접 수정할 수 있도록 정책 테이블로 관리하여, 정책 변경 시 코드 배포 없이 운영 가능합니다.
  • QR 디퍼드 딥링크: 외부 솔루션 도입 없이 자체 구현으로 진행합니다. 디자이너 ID를 담은 QR URL을 만들어, 앱 설치 시 Android Install Referrer API(공식·무료)로 자동 매칭하고, iOS는 Universal Link + 첫 실행 환영 화면 + 휴대폰 번호 매칭으로 커버합니다. 솔루션 비용·서비스 종료 리스크(Firebase Dynamic Links 종료 사례) 회피.
  • 커머스 정산: 결제 완료 시 디자이너 등급별 차등 마진(4:6 / 5.5:4.5 / 6:4)에 따라 디자이너 수익과 플랫폼 수수료를 자동 기록하고, 월 단위로 디자이너 송금 가능 잔액을 집계합니다. 결제 시점 등급 스냅샷을 저장하여 등급 변경 시 소급 적용을 방지합니다.

3. 인스타그램형 단골 피드 메인 — 유사 매칭 플랫폼 화면 노하우 직접 활용

대면 미팅에서 확인된 메인 화면(상단 단골 디자이너 아바타 카드 + 하단 디자이너 작업 사진 쓰레드 피드) 구조는 디지털 명함 매칭 플랫폼에서 구현한 "근거리 추천 사용자 + 비즈니스 소셜 피드" 화면 구조와 동일 패턴입니다. 가로 스크롤 아바타 리스트, 무한 스크롤 피드 페이지네이션, 이미지 캐싱 및 프리페치(Flutter cached_network_image + flutter_bloc 스트림 처리), 좋아요·예약 즉시 전환 CTA 등 핵심 구현 패턴을 그대로 재사용하여 메인 화면을 안정적으로 빠르게 완성합니다.

4. 디자이너 간 데이터 격리 — RLS + 멀티테넌트 모델

모든 고객 데이터(예약, 처방전, 쿠폰, 포인트)는 designer_id 외래키를 가지며, PostgreSQL Row-Level Security(RLS) 정책으로 디자이너 본인 데이터만 조회·수정 가능하도록 DB 레벨에서 강제합니다. 애플리케이션 코드 버그가 있어도 다른 디자이너의 데이터 노출이 원천 차단됩니다. 권한은 CASL 기반 RBAC로 디자이너/관리자/슈퍼관리자 역할 분리.

5. 관리자 페이지 — B2B2C SaaS 운영 콘솔 패턴 재활용

본 프로젝트와 동일한 B2B2C 구조의 헬스케어 SaaS(시니어 주간보호센터 통합 관리 플랫폼)에서 구축한 운영자 콘솔 구조를 재활용합니다. 사이드바 네비게이션 + 통계 카드 + 가입 승인 워크플로우 + 정산 잔액 대시보드 + 감사 로그 패턴이 본 프로젝트의 디자이너 가입 승인·정산·포인트 부채 관리 요구사항과 1:1 매칭됩니다.

6. Flutter 시연 코드 활용 — 폐기 없이 점진 고도화

제공된 단일 main.dart 시연 코드를 분석하여 (1) 화면별로 feature 디렉토리 분리, (2) Riverpod Provider/AsyncNotifier 패턴으로 상태관리 도입, (3) Repository 레이어 추가로 API 연동 준비, (4) 누락된 화면 30%를 디자인 가이드 기준으로 보완. 기존 UI는 최대한 보존하여 기획자가 검토한 화면 흐름이 그대로 유지됩니다.

7. 미확정 정책 사전 협의 — 1주차 정책 결정 워크숍

15분 웨이팅 알림 자동·수동 발송 여부, 환불 정책 세부 시점, 위약금 산정 기준, 앱 구조(옵션 A/B) 선택 등 미확정 항목을 1주차 정책 결정 워크숍에서 일괄 협의·확정합니다. 정책 결정 사항은 정책 문서로 정리되어 개발 명세서와 함께 단일 진실 공급원(Single Source of Truth)으로 관리됩니다.

2.1 양방향 웨이팅 시스템 (명세서 5.1, 11.3)

작동 흐름 (사용자 관점)

  1. 고객이 마감 시간대 슬롯에 웨이팅 신청 → 대기열에 등록 (FIFO 순서)
  2. 예약자가 취소 발생 → 대기열 1순위 고객에게 15분 타임어택 푸시 전송
  3. 1순위 고객이 15분 내 결제 완료 → 슬롯 확정, 대기열에서 제거
  4. 1순위 고객이 15분 내 결제하지 않음 → 자동으로 2순위로 이전, 15분 타이머 다시 시작
  5. 대기열이 소진되거나 누군가 결제 완료할 때까지 반복

기술 구현 방식

  • 대기열 저장: Redis FIFO 큐(LIST 자료구조)에 사용자 ID를 순서대로 저장. 푸시·자동 이전 시 1순위를 원자적으로 추출(LPOP). Redis 특성상 큐 read/write 속도가 빠르고 순서 보장이 명확
  • 결제 확정 시 DB 기록: 사용자가 결제 완료하면 PostgreSQL reservations 테이블에 예약 INSERT + 대기열에서 해당 사용자 제거를 단일 트랜잭션으로 처리. 슬롯 점유는 결제 확정 시점에 DB에 영구 기록
  • 15분 타이머: Redis TTL 키 활용. 1순위에게 푸시 발송 시 SET active_waiting:slot_id user_id EX 900 (900초). TTL 만료 이벤트 감지(Redis Keyspace Notifications) 또는 1분 주기 폴링으로 만료된 슬롯을 자동으로 다음 사용자에게 이전
  • 다중 사용자 동시 결제 시도 동시성 제어: 결제 확정 트랜잭션에서 PostgreSQL SELECT FOR UPDATE로 슬롯 행 잠금. 동시에 결제 버튼을 눌러도 트랜잭션이 순차 처리되어 단일 고객만 통과
  • 중복 요청 방어: 결제·예약 수락 API에 멱등성 키(Idempotency-Key) 헤더 적용. 네트워크 재시도·중복 클릭에도 단일 처리 보장
  • 상태 추적: 큐 등록 → 푸시 발송 → 수락/거절/만료 → 결제 완료 전 과정을 audit_logs에 기록 (분쟁 대응 및 정합성 검증용)
  • 알림: FCM 푸시 기본, 앱 비활성 사용자는 알림톡 fallback
  • 시간 기준: AWS/NCP 서버는 NTP가 자동 동기화되어 있어 서버 시각 기준으로 타이머 운영. 클라이언트 앱 타이머는 서버 응답의 절대 시각을 기준으로 동기화하여 클라이언트 시계 변조에 대비

자동/수동 발송 정책 제안

  • 주간(08:00~22:00): 자동 발송 — 취소 발생 즉시 1순위에게 푸시
  • 야간(22:00~08:00): 수동 발송 권장 — 사용자가 새벽에 푸시 받는 부담을 줄이기 위해, 디자이너가 다음 날 오전 확인 후 수동으로 발송 트리거
  • 위 기본값은 1주차 정책 워크숍에서 협의 후 확정

  • 환불 실패 시 재시도 큐 등록(짧은 backoff, 3회 시도). 최종 실패 시 운영자 콘솔 알림

  • 포인트 1년 만료: 발급 후 1년 경과 시 자동 만료 처리 (cron job). 만료 시점에 회사 부채 → 매출 환입으로 회계 처리
  • 관리자 콘솔에 미사용 포인트 부채 잔액 실시간 대시보드 제공 (명세서 11.6 반영)

  • 분쟁 처리: 디자이너↔고객 분쟁 시 운영자 콘솔에서 수동 환불 권한 + 분쟁 이력·증빙 기록 워크플로우
  • 환불 영수증·세금계산서: PortOne이 환불 영수증 자동 발급, 세금계산서는 디자이너 사업자 정보 기반 자동 또는 수동 워크플로우

2.3 디자이너 마케팅 포인트 시스템 (명세서 5.13)

정산 가속 + 환영 포인트 통합 잔고 — 적립·사용·만료 트랜잭션 무결성

  • 모든 포인트 적립·사용·만료·환불을 거래 내역 테이블에 단일 트랜잭션으로 기록 — 거래 도중 실패 시 전체 롤백으로 잔고 정합성 보장
  • 잔액은 거래 내역 합산으로 도출하며, 자주 조회되는 잔액은 캐싱(Redis)으로 빠르게 응답
  • 정산 가속 포인트·환영 포인트는 통합 잔고로 표시하되, 출처별 분리 저장 (회계·만료 처리·세무 분기를 위해)
  • 매일 자정 reconciliation으로 거래 합계 = 잔액 일치 여부 자동 검증

FIFO 사용 (만료 임박 우선)

  • 포인트 사용 시 만료일이 빠른 순서로 자동 차감 (FIFO 차감 로직)
  • 만료 30일 전·7일 전 사전 알림 (푸시 + 앱 비활성 시 알림톡 fallback)
  • 만료 처리는 매일 자정 cron job으로 자동 진행

회계 카테고리 분기 (CCAC vs 매출 차감 vs 마케팅 비용)

  • 포인트 발급·사용·만료의 회계 분류는 출처별로 다르게 처리
  • 환영 포인트(신규 가입 시): CCAC (고객 획득 비용)로 분류 — 마케팅 비용 처리
  • 정산 가속 포인트: 매출 차감으로 분류 — 디자이너 매출에서 직접 차감
  • 이벤트·프로모션 포인트: 마케팅 비용으로 분류
  • 거래 내역 테이블에 회계 카테고리 컬럼을 두어 자동 분류 → 월간 회계 보고서에 카테고리별 자동 집계

환영 포인트 어뷰징 방지 (동일 IP·기기·휴대폰)

  • 동일 IP에서 단기간 다중 가입 차단 (IP + 시간 윈도우 임계값)
  • 디바이스 ID 핑거프린팅으로 동일 기기 다중 계정 가입 차단
  • 본인인증 휴대폰 번호 1회 1계정 제약 (본인인증 PG에서 자동 차단)
  • 의심 가입 패턴 자동 운영자 검토 큐 등록
  • 적발 시 활성 포인트 차감 + 계정 정지 + audit_logs 기록

월 한도 20명 + 마일스톤 단가 자동 상승

  • 디자이너가 단골 등록할 수 있는 월 한도(예: 20명)를 정책 테이블로 관리 → 운영자가 운영 중 조정 가능
  • 마일스톤 단가 자동 상승: 디자이너의 누적 단골 수에 따라 다음 단골 등록 시 적립되는 환영 포인트 단가가 자동으로 상승하는 룰을 코드로 정의 (예: 1~10명 단가 A, 11~20명 단가 B)

2.4 정산 투명화 시스템 (명세서 5.11)

실시간 거래 피드 — SSE 또는 폴링 권장

  • 본 거래 피드는 서버에서 클라이언트로의 단방향 통신만 필요하므로 SSE(Server-Sent Events) 또는 폴링 방식을 권장합니다
  • WebSocket은 양방향 통신이 필요한 채팅·실시간 협업 같은 케이스에 사용 — 본 서비스에는 불필요한 복잡도
  • SSE는 HTTP 기반이라 로드밸런서·프록시 호환성이 좋고 자동 재연결이 내장되어 안정적
  • 초기에는 10~30초 주기 폴링으로 시작하고, 트래픽·요구사항에 따라 SSE로 전환하는 방향도 가능

거래 발생 → 1초 이내 디자이너 화면 반영

  • 거래 발생 즉시 디자이너에게 FCM 푸시 전송 (보통 100ms~1초 이내 도달)
  • 앱이 푸시 수신 시 자동으로 거래 내역 화면 갱신
  • 앱이 활성 상태에서는 SSE 또는 짧은 주기 폴링으로 거의 실시간 반영
  • 1초 이내 SLA가 강하게 필요한 경우 SSE 적용을 권장

매일 자정 reconciliation — 거래 합계 vs 디자이너 잔액 일치 검증

  • 매일 자정 cron job으로 자체 DB 내 거래 내역 합계와 디자이너별 잔액 컬럼의 일치 여부 검증
  • 거래(매출·환불·위약금·송금) 합계 = 디자이너 잔액 공식이 깨지면 데이터 무결성 오류이므로 즉시 운영자 콘솔 알림 + Slack 알림 + audit_logs 기록
  • 본 reconciliation은 자체 회계 데이터의 무결성 검증입니다 (결제사와의 외부 매칭이 아니라 자체 DB 내부의 거래 합계 vs 잔액 검증)

RLS — 보안 강화 옵션 (A·B) 선택 시 적용

  • 현재 견적(4,250만원)은 베이스라인 기준: 애플리케이션 레이어(ORM 인터셉터)에서 designer_id를 강제로 격리하는 방식
  • 명세서가 MVP 요건으로 규정한 PostgreSQL Row-Level Security 정책 적용은 견적서의 [보안 강화 옵션 A·B] 선택 시 포함됩니다 (RLS 8 M/D 추가)
  • 옵션 선택 시 적용 내용:
    • 모든 디자이너 종속 테이블(예약·처방전·쿠폰·포인트·단골관계·매출)에 designer_id 기반 RLS 정책 적용

2.5 디자이너 미활동 차단 메커니즘 (명세서 5.12)

트리거 A·B 일별 배치 검사 — 정확도 + 성능

  • 매일 자정 cron job으로 디자이너 테이블의 마지막 활동 일자(last_active_at)·마지막 매출 발생 일자(last_revenue_at)를 인덱스 기반으로 조회하여 트리거 조건 검사
  • 인덱스 적용으로 수만 명 규모에서도 수 분 내 완료 가능
  • 트리거 조건 충족 시 사전 알림 단계로 진입, 차단 임박 단계별로 상태 업데이트

사전 알림 4채널 (푸시·알림톡·SMS·이메일)

  • 차단 임박 시점(예: D-30, D-7, D-1)별로 단계적 알림 발송
  • 일반 단계: 푸시(앱 활성) → 알림톡(앱 비활성) → SMS → 이메일 우선순위 fallback
  • 긴급 임계 시점(D-1 등): 4채널 동시 발송으로 도달률 극대화
  • 발송 결과는 audit_logs 기록 (도달률 모니터링·클레임 대응)

AWS KMS 키 회전 + 디자이너별 단골 데이터 암호화 — 보안 강화 옵션 (A·B) 선택 시 적용

  • 현재 견적(4,250만원)은 베이스라인 기준: 차단 디자이너 단골 데이터를 AES-256으로 암호화하되, 키는 환경 변수 + AWS Secrets Manager로 관리
  • 명세서가 MVP 요건으로 규정한 AWS KMS 키 회전 적용은 견적서의 [보안 강화 옵션 A·B] 선택 시 포함됩니다 (KMS 공통 5.5 M/D + 옵션 A는 90일 자동/옵션 B는 30일 on-demand 2 M/D 추가)
  • 옵션 선택 시 적용 내용:
    • 차단 디자이너 단골 데이터는 별도 격리 영역에 보관, AES-256 + AWS KMS 관리 키로 컬럼 암호화
    • 옵션 A (90일 자동회전): AWS KMS 기본 자동 키 회전 기능 활용 (별도 자동화 구현 불필요)
    • 옵션 B (30일 on-demand): 30일 주기 on-demand 키 회전 자동화 — Lambda 스케줄러 + 롤오버 검증 자체 구현
    • 키 사용 이력은 CloudTrail로 자동 기록 (감사 추적)

복합 인증 자동 검증 (재활성화 조건)

  • 차단 해제 조건(예: 골드 재결제 + 1개월 이내 단골 시술 1건)을 코드 룰로 정의
  • 디자이너가 활동 재개 신청 시 자동으로 조건 검증
  • 결제 이력은 PortOne 거래 데이터 기반, 단골 시술 이력은 reservations 테이블 기반으로 검증
  • 조건 미충족 시 운영자 검토 워크플로우로 우회 가능

명시적 휴업 신고 — 자동 차단 우회 + 즉시 복원

  • 디자이너 앱에 "휴업 신고" 기능 제공 (기간 선택 또는 무기한)
  • 휴업 신고 시 자동 차단 대상에서 제외 (휴업 상태로 별도 분류)
  • 휴업 기간 종료 또는 디자이너가 직접 휴업 해제 시 즉시 정상 상태로 복원
  • 휴업 이력은 audit_logs 기록

2.6 디퍼드 딥링크 (QR 단골 자동 매칭) (명세서 5.6)

솔루션 선택 — 자체 구현 제안

외부 솔루션(Branch.io 등) 도입 없이 자체 구현으로 진행 가능하다고 판단됩니다. 자체 QR + Universal Link/App Link + Android Install Referrer + 휴대폰 번호 매칭 조합으로 본 서비스가 필요로 하는 매칭 정확도를 확보할 수 있고, 외부 솔루션 종속성·운영 비용·서비스 종료 리스크(Firebase Dynamic Links 2025.08 종료 사례)도 회피할 수 있습니다. 향후 MAU·매칭 손실률 추이를 보고 외부 솔루션 도입을 재검토하는 방향이 합리적입니다.

구현 아키텍처

  1. QR 생성
    • 디자이너 가입 시 고유 식별자(slug) 발급
    • QR URL 형식: https://mydesigner.com/d/{slug}?utm_source=qr...
    • UTM 파라미터로 유입 분석 지원
    • QR 만료 시간(예: 90일) + 인증된 디자이너만 생성 + 사용 추적 audit_logs
  2. 딥링크 도메인 검증 파일 호스팅 (필수)
    • iOS: apple-app-site-association 호스팅 (Universal Link 작동 조건)
    • Android: assetlinks.json 호스팅 (App Link 작동 조건)
  3. URL 라우팅
    • 앱 설치 + 일반 브라우저: Universal Link/App Link 자동 작동 → 앱이 designer_id 받아 매칭
    • 앱 미설치: Play Store / App Store 다운로드 페이지 리디렉트
    • 카카오톡 인앱 브라우저: User-Agent 감지 → 폴백 페이지에서 "Chrome/Safari로 열기" 안내. Android는 intent:// 스킴으로 외부 브라우저 직접 호출
  4. 디퍼드 매칭 (앱 미설치 → 설치 후 매칭)
    • Android: Google Play Install Referrer API (공식·무료, 정확도 99%+). Play Store URL에 designer_id가 referrer로 자동 첨부되며, 앱 첫 실행 시 자동 수신하여 매칭
    • iOS (다중 폴백 전략)
      • (a) 휴대폰 번호 매칭: 디자이너가 매장에서 단골 고객의 휴대폰 번호를 사전 등록 → 사용자가 가입 시 휴대폰이 일치하면 자동 단골 매칭. 비즈니스 흐름에 자연스럽게 통합되며 정확도가 높음
      • (b) 환영 화면 추천 코드 입력 — 앱 첫 실행 시 "어떤 디자이너의 추천으로 오셨나요?" 화면 + 디자이너 검색
      • (c) Apple Search Ads API (광고 캠페인 한정 사용)
  5. 매칭 정확도 모니터링
    • 매칭 성공/실패 로그를 운영자 콘솔 대시보드에 표시
    • 매칭 손실률이 일정 임계값을 초과하면 외부 솔루션 도입을 재검토
    • 디자이너별 QR 유입 통계 제공
  6. 예상 매칭 정확도: Android 99%+ / iOS 95% 내외 (휴대폰 매칭 + 환영 화면 폴백 포함)
  7. 향후 솔루션 도입 검토 시점: MAU 증가에 따른 iOS 매칭 손실률이 비즈니스 손해로 누적되는 시점에 Branch.io 또는 Adjust 도입 재검토 (자체 구현 폴백은 유지)

  • 디자이너 본인에게 "관리자가 회원님 데이터를 조회했습니다" 알림 발송 (앱 푸시 + 알림톡)

  • 슈퍼관리자 접근 시 사유 입력 의무화 + audit_logs 강화 기록 (조회한 데이터 범위까지 상세 기록)
  • 분기별 슈퍼관리자 접근 이력 검토 워크플로우 제공 (감사 추적)
